# Implementation Plan — 2026-05-22 (rev. 2 after codex review)

Status: **rev. 2 after the first `codex exec` architectural review**. Changes
since rev. 1:

- HDF5 in-memory: drop the `NamedTempFile` shim — `hdf5-reader` 0.5.0 already
  exposes `Hdf5File::from_bytes` / `from_storage` and `netcdf-reader` 0.5.0
  exposes `NcFile::from_bytes_with_options`. The sidecar resolver routes the
  bytes directly into those constructors.
- `open_path` is **not** collapsed in M1. `read_path` and `read_bytes` stay
  as parallel entry points; the resolver is only used when bytes are
  decoded. Eliminating one of the two paths is deferred until every
  path-only reader (Parquet, plus any survivor) is converted.
- Sidecar scope expanded to:
  - MATLAB Indian Pines: `indian_pines_gt.mat` sidecar
    (`readers/matlab.rs:794`).
  - NetCDF ARM MFRSR: optional `.yaml` QC sidecar
    (`readers/netcdf.rs:1777`).
  - HDF5 external files / external links: routed through new
    `ExternalFileResolver` / `ExternalLinkResolver` impls backed by
    `SidecarResolver`.
- M2 reframed as **independent of M1**; tolerances become per-format,
  per-signal `abs <= max(abs_tol, rel_tol * scale)` formulas with explicit
  documentation, not flat constants.
- M3 must run **before M2** to avoid invalidating JCAMP goldens and
  conformance baselines mid-flight.
- M4 wheels: no preventive HDF5/Linux-only gating. Wheels build full
  features on every platform; downgrades happen only after a documented
  per-OS failure.
- File lists updated to include `core/src/error.rs`,
  `crates/nirs4all-formats/src/lib.rs` re-exports, the R `NAMESPACE`,
  `docs/DATA_MODEL.md`, `docs/FORMAT_MATRIX.md`, M2's
  `tests/conformance/pyproject.toml`, M4's `bindings/python/Cargo.toml` and
  `pyproject.toml`, and the `nirs4all-formats-capi` build deps.

The four milestones, in execution order:

1. **M1** — Sidecar resolver for `Reader::read_bytes` (item 5 of
   `STATUS.md` "Next Agent Prompt").
2. **M3** — JCAMP heterogeneous `LINK` API + strict checkpoint
   hardening (item 6). Moved before M2 to freeze goldens.
3. **M2** — External reference-reader conformance for OPUS, SPC, JCAMP,
   SED, SIG, ASM, HDF5 (item 4).
4. **M4** — Phase 6 release pipelines (PyPI wheels, C ABI archive, R
   source build) per `ROADMAP.md`.

Out of scope (explicit non-goals, all already documented as blocked or
sourcing tasks):

- new native readers for Foss/Perten/Metrohm/PP Systems/ASD calibration/Vision
  Air/Shimadzu native/VIAVI `.pri` (blocked by missing fixtures);
- BUCHI redistributable non-null target, OMNIC `.srsx` (sample-blocked);
- AVIRIS / Indian Pines redistribution decision (legal/policy);
- NEON/Specim/HySpex/Headwall cube sourcing;
- Allotrope ADF full ontology/SDK conformance (no redistributable corpus).

Each milestone ends on a clean green-gate sequence identical in spirit to the
one in `STATUS.md`. No milestone is marked complete until every command in its
green gate passes locally.

## Cross-cutting rules

- **Rust is the source of truth.** All format logic lives in
  `crates/nirs4all-formats/src/readers/*`. Bindings stay thin and only wire data
  out.
- **No backward-compat shims.** When the Reader trait changes, every
  implementation moves at once; no deprecated methods retained.
- **Real fixtures only.** Each milestone has an explicit *Tests vs real data*
  subsection naming the committed files exercised. Synthetic fixtures are
  used only for adversarial paths.
- **Green gate at the end** = the full command sequence from
  `STATUS.md § Last Green Gate`, plus the new tests this milestone adds.
- **Docs updated together.** `STATUS.md`, `ROADMAP.md`, `FORMAT_MATRIX.md`,
  `IMPLEMENTATION_DASHBOARD.md` and `README.md` are touched in the same
  commit as the code, not in a follow-up.

## M1 — Sidecar resolver for `Reader::read_bytes`

### Why now

Today `open_bytes`/`Reader::read_bytes` work for every single-file reader, but
ENVI Standard, AVIRIS ERDAS LAN, FGI HDF5+XML, MATLAB v7.3, MATLAB Indian
Pines ground-truth and the ARM MFRSR QC YAML sidecar still require a real
filesystem because they call `std::fs::read` / `Hdf5File::open(path)` /
`File::open(path)` directly. Bindings (PyO3, WASM, R) hit this limitation and
`crates/nirs4all-formats/tests/open_bytes.rs` already encodes the gap as
expected-error assertions.

### Design

#### New core trait

In a new submodule `crates/nirs4all-formats-core/src/sidecar.rs`:

```rust
/// Resolves files that a reader needs in addition to its primary payload.
/// Implementations may be backed by a real directory or an in-memory map.
pub trait SidecarResolver: Send + Sync {
    /// Returns the raw bytes for a sidecar referenced by `relative` path.
    /// The path is resolved relative to the primary file's logical parent.
    fn read(&self, relative: &Path) -> Result<Vec<u8>>;

    /// Returns true iff `read(relative)` would succeed. Used for optional
    /// sidecars (e.g. the AVIRIS `.GIS` ground-truth file).
    fn contains(&self, relative: &Path) -> bool;

    /// Optional list of every known sidecar key, used by debugging tools
    /// and the CLI dump. Empty by default.
    fn list(&self) -> Vec<PathBuf> {
        Vec::new()
    }
}
```

A new error variant in `crates/nirs4all-formats-core/src/error.rs`:

```rust
#[error("sidecar '{path}' not available: {reason}")]
UnsupportedSidecar { path: PathBuf, reason: String },
```

Two concrete impls in a new module
`crates/nirs4all-formats/src/sidecars.rs`:

- `FsSidecars { base: PathBuf }` — reads relative paths under `base` via
  `std::fs::read`. Constructed from `path.parent()`.
- `InMemorySidecars { map: BTreeMap<PathBuf, Vec<u8>> }` — pure in-memory.
  Public constructor `InMemorySidecars::new(map)` plus a builder
  `.with(path, bytes)` for ergonomics.
- `NoSidecars` — zero-sized impl that errors with `UnsupportedSidecar`
  on every `read` / `contains`. Used by the default `Reader::read_bytes`
  shim so old call sites still error explicitly when a real resolver was
  required.

Path normalization: keys are case-sensitive on every OS in the trait
itself. Format readers do their own case fallback when they need to (ENVI
is the only known case-mixing format and it already iterates over a
candidate-extension list).

#### Reader trait extension

In `crates/nirs4all-formats/src/registry.rs`:

```rust
pub trait Reader: Send + Sync {
    // existing methods unchanged: name, sniff, read_path,
    // read_path_with_options, read_bytes, read_bytes_with_options.

    /// Decode an in-memory primary payload using a sidecar resolver.
    /// Default implementation delegates to `read_bytes_with_options`; the
    /// default ignores the resolver, which means single-file readers
    /// automatically work with any resolver.
    fn read_bytes_with_sidecars(
        &self,
        name: &Path,
        bytes: &[u8],
        sidecars: &dyn SidecarResolver,
        options: &ReadOptions,
    ) -> Result<Vec<SpectralRecord>> {
        let _ = sidecars;
        self.read_bytes_with_options(name, bytes, options)
    }
}
```

`open_path*` keeps its current implementation. **`open_path` is not
collapsed.** Instead, sidecar-bearing readers gain a small `read_inner`
function that takes a `&dyn SidecarResolver` and is called from both
`read_path` (with `FsSidecars`) and `read_bytes_with_sidecars` (with the
caller-supplied resolver). This keeps risk localized: readers that don't
need sidecars are not touched.

Two new public registry entry points:

```rust
pub fn open_with_sidecars(
    name: impl AsRef<Path>,
    bytes: &[u8],
    sidecars: &dyn SidecarResolver,
) -> Result<Vec<SpectralRecord>>;

pub fn open_with_sidecars_and_options(
    name: impl AsRef<Path>,
    bytes: &[u8],
    sidecars: &dyn SidecarResolver,
    options: &ReadOptions,
) -> Result<Vec<SpectralRecord>>;
```

`open_bytes_with_options` keeps its current behavior: it calls
`read_bytes_with_options` with the `NoSidecars` resolver implicit through
the default trait shim, so sidecar-bearing readers still error there. The
new entry point is the way to decode them in-memory.

#### Per-reader refactors

1. **ENVI SLI / ENVI Standard** (`readers/envi_sli.rs`)
   - Replace every `std::fs::read(&header_path)` /
     `std::fs::read(&data_path)` with `sidecars.read(rel)`.
   - `paired_paths` and `resolve_data_path` return relative paths.
   - The path-mode entry point builds an `FsSidecars` from
     `header_path.parent()` and reuses the same inner function.

2. **ERDAS LAN** (`readers/erdas_lan.rs`)
   - `<base>.spc` axis sidecar + optional fixed-name `92AV3GT.GIS` go
     through the resolver. The primary `.lan` payload arrives as bytes
     directly.

3. **FGI HDF5+XML** (`readers/fgi_xml.rs`)
   - `<DataReference path="...">` resolves through the sidecar trait.
   - The HDF5 sidecar bytes feed into `Hdf5File::from_bytes` (see HDF5
     section below).

4. **MATLAB Indian Pines** (`readers/matlab.rs`)
   - `indian_pines_gt.mat` ground-truth sidecar at
     `matlab.rs:794` switches to `sidecars.read(...)`. The `read_path`
     side keeps using `FsSidecars`.

5. **NetCDF ARM MFRSR** (`readers/netcdf.rs`)
   - The QC YAML sidecar at `netcdf.rs:1777` goes through the resolver
     with `sidecars.contains(...)` for the optional probe.

6. **HDF5 / NetCDF / MATLAB v7.3 / Allotrope ADF in-memory decoding**
   - `hdf5-reader` 0.5.0 already exposes
     `Hdf5File::from_bytes(data: &[u8])`, `from_vec(Vec<u8>)`,
     `from_bytes_with_options`, and `from_storage(DynStorage)` (see the
     dependency's `src/lib.rs:384-418`). Use these directly. No
     `NamedTempFile`.
   - `netcdf-reader` 0.5.0 exposes
     `NcFile::from_bytes_with_options` (`src/lib.rs:223`). Same.
   - For HDF5 files with external links/files, install
     `Arc<dyn ExternalFileResolver>` and
     `Arc<dyn ExternalLinkResolver>` in `OpenOptions` (per
     `hdf5-reader/src/lib.rs:79`). Both impls are thin wrappers that
     forward to our `SidecarResolver`:

     ```rust
     struct SidecarBackedExternal { sidecars: Arc<dyn SidecarResolver> }
     impl ExternalFileResolver for SidecarBackedExternal { /* ... */ }
     ```

     `SidecarResolver` is therefore consumed via `Arc` in the readers, so
     the trait stays object-safe and shareable across threads.

   - Helper `open_hdf5(name: &Path, bytes: &[u8], sidecars: &dyn SidecarResolver)`
     in `readers/util.rs` builds `OpenOptions`, wires the external
     resolvers and returns a `Hdf5File`.

#### Parquet path-only reader

`ParquetReader` currently uses `File::open(path)` (`readers/parquet.rs:43`).
Out of M1 scope but documented in a follow-up TODO: convert to
`bytes::Bytes`-based reading. This is the only remaining path-only reader
after M1; collapsing `open_path` becomes safe in a later milestone.

#### CLI

`crates/nirs4all-formats-cli/src/main.rs` gains an optional repeated
`--sidecar key=path` argument for the `read-json` and (new) `read-bytes`
subcommands. Sidecars load via `std::fs::read` and assemble into an
`InMemorySidecars`. Used in tests and ad-hoc validation.

#### Python binding

`bindings/python/src/lib.rs` adds:

```python
def open_with_sidecars(
    name: str | os.PathLike,
    bytes_: bytes,
    sidecars: Mapping[str, bytes],
    *,
    rows=None, cols=None, pixels=None,
) -> list[dict]:
```

Implementation: build `InMemorySidecars` from the dict, then call
`open_with_sidecars_and_options`.

#### R binding

`bindings/r/nirs4allformats/R/io.R` gains
`nirs4allformats_open_with_sidecars(name, raw_bytes, sidecars = list(...))`,
exported in `bindings/r/nirs4allformats/NAMESPACE`. Backed by the same registry
call through the static extendr lib (with CLI subprocess fallback writing
each sidecar to a temporary directory).

#### WASM binding

`bindings/wasm/src/lib.rs` adds
`openWithSidecars(filename: string, primary: Uint8Array,
sidecars: Record<string, Uint8Array>)`. In M1 this entry point covers

The existing "C libraries" comment in `bindings/wasm/src/lib.rs:36` is

`openWithSidecars` is a follow-up ticket (not part of M1).

## M3 — JCAMP heterogeneous `LINK` API + strict checkpoint hardening

### Why now (and why before M2)

`STATUS.md` item 6 calls for both a public API decision on heterogeneous
`LINK` children and continued hardening of `XYDATA` line-start X
checkpoints. The current implementation already emits multiple records for
`LINK`/multi-block files but does not expose a stable identifier for the
parent grouping.

M3 lands **before M2** because the JCAMP conformance harness in M2 will
compare nirs4all-formats records to `jcamp` (Python). If we add `LINK`
metadata after M2, every JCAMP conformance fixture has to be
re-baselined and the goldens regenerated twice.

### Design

1. **Record metadata for LINK children**: each record gains
   `metadata["link_parent_id"]`, `metadata["link_index"]`,
   `metadata["link_total"]`. The parent_id is a stable hash of the
   `##TITLE` of the parent block + the file's SHA256 prefix.
2. **Heterogeneous LINK**: a `LINK` block whose children have different
   axis units / kinds returns one record per child with an explicit
   `metadata["link_relation"]` ∈
   `{"sample", "dark", "reference", "unknown"}` inferred from
   `##DATA TYPE=` and known Ocean Optics block titles.
3. **Strict checkpoints**: replace the current flat
   `jcamp_xydata_x_checkpoint_mismatch` warning with a structured
   warning `jcamp_xydata_x_checkpoint_drift:line=<n>:abs=<v>:rel=<v>`
   that reports both absolute and relative deltas.
4. **PEAK TABLE / PEAK ASSIGNMENTS**: no real public fixture is
   committed; the existing synthetic-only path is documented as the
   contract until a real fixture lands.

### Tests vs real data

- `samples/jcamp_dx/Ocean_Optics_link.dx` — multi-block LINK
- `samples/jcamp_dx/nmr_ntuples.dx` — NMR NTUPLES
- `samples/jcamp_dx/peak_assignments.dx` — sparse PEAK ASSIGNMENTS
- Plus every other committed `.dx`/`.jdx` JCAMP fixture — they must
  acquire the new `link_*` metadata (or stay unchanged when no LINK is
  present).

Goldens under `tests/goldens/jcamp_dx/*` regenerate via the documented
`NIRS4ALL_FORMATS_ACCEPT_GOLDENS=1` workflow; the PR commit message captures
the before/after delta.

### Files modified

- `crates/nirs4all-formats/src/readers/jcamp.rs`
- `crates/nirs4all-formats/tests/readers.rs` (jcamp section)
- `crates/nirs4all-formats/tests/goldens/jcamp_dx/*` (regenerated)
- `docs/formats/jcamp-dx.md`
- `docs/STATUS.md`, `docs/FORMAT_MATRIX.md`,
  `docs/IMPLEMENTATION_DASHBOARD.md`

### M3 Green Gate

Standard sequence + `cargo test -p nirs4all-formats --test goldens` + diff
review on the regenerated golden JSONs.

Estimated effort: 0.5–1 day.

## M2 — External reference-reader conformance

### Why now

`docs/CONFORMANCE.md` already names the target reference readers
(`opusreader2`, `brukeropus`, `spc-spectra`, `jcamp`, `spectrolab`,
`spectrochempy`, `allotropy`, `h5py`). The current `conformance.yml` is a
placeholder. Closing this gap unlocks Beta status for the seven readers
and gives downstream users a reliable equivalence claim. Runs after M3
so the JCAMP baseline is final.

### Design

A single Python harness under `tests/conformance/`, invoked with
`pytest -m conformance`. Marker keeps it out of the default `cargo test`
flow; the marker is registered in `tests/conformance/pyproject.toml` and
`tests/conformance/conftest.py`. Each format has its own test module that:

1. Discovers every committed fixture for that format under `samples/<dir>/`.
2. Loads the same file through `nirs4all_formats.open_path` and through the
   reference reader.
3. Normalizes both outputs to a small comparable dict
   (`axis: list[float]`, `values: list[float]`, optional metadata
   subset).
4. Asserts equivalence with the per-format tolerance formula.

#### Tolerance model

Per-format, per-signal, per-reference-reader tolerance is

```
ok = abs(diff) <= max(abs_tol, rel_tol * abs(reference_value))
```

with axis and value scales handled separately. The pair `(abs_tol,
rel_tol)` is documented in `tests/conformance/tolerances.toml`. Examples:

- SED / SIG axes: `abs_tol=0` and `rel_tol=1e-6` because both nirs4all-formats
  and `spectrolab` parse the same ASCII text (`readers/sed.rs:74`,
  `readers/svc_sig.rs:100`); any difference is a parsing bug.
- SPC axes: `abs_tol=1e-6` and `rel_tol=1e-7` to cover the explicit-X
  float32 round-trip (`readers/galactic_spc.rs:604`).
- OPUS axes (constructed from `FXV/LXV/NPT` at
  `readers/bruker_opus.rs:442`): `abs_tol=0` and
  `rel_tol=1e-12` — generated linearly; must match bit-for-bit modulo
  arithmetic order.
- OPUS values: `abs_tol=1e-7` and `rel_tol=1e-6` after applying the
  `CSF` scale at `readers/bruker_opus.rs:515`. Where opusreader2 applies
  a different gain or sign convention, the fixture is marked `xfail`
  with the reason inline in `tolerances.toml` — never a flat 1e-4
  cover.
- JCAMP ASDF-decoded values: `abs_tol=0` and `rel_tol=1e-12` (pure
  integer decoding); the original ASCII fixed-point values are exact.
- HDF5: bit-for-bit `abs_tol=0` and `rel_tol=0` against `h5py`.

## M4 — Phase 6 packaging

### Why now

`ROADMAP.md § Phase 6` is the last item gating a public release. Today
there is no `release.yml`, no `cbindgen`, no published wheel.

### Design

#### Python wheels via `cibuildwheel`

`.github/workflows/release.yml`:

- triggered on `v*` tags + `workflow_dispatch` for dry-runs;
- matrix: Linux (manylinux2014 x86_64, aarch64), macOS (x86_64, arm64),
  Windows (x86_64);
- **full features on every platform** (the pure-Rust hdf5-reader /
  netcdf-reader make this safe — see `hdf5-reader/Cargo.toml:23` and
  `netcdf-reader/Cargo.toml:23` which both declare "no C dependencies");
- per-job: `pip install cibuildwheel`, run `cibuildwheel
  bindings/python --output-dir wheelhouse`;
- aggregate job: `pypa/gh-action-pypi-publish@release/v1` with OIDC
  trusted publishing (no API token);
- sdist built once via `maturin sdist`;
- the dry-run flow uploads to TestPyPI first; the tagged-release flow
  uploads to PyPI.

If a platform fails for a documented reason (e.g. a future C-backed
feature), the workflow downgrades that one wheel via an explicit
`--features` exclusion and writes the capability matrix into
`docs/RELEASE.md` so consumers know what each wheel supports.

`bindings/python/Cargo.toml` keeps default features on. A new
`bindings/python/pyproject.toml` `[tool.cibuildwheel]` section pins:

- `build-frontend = "build"`
- `before-all = ["rustup default stable"]`
- `test-command = "pytest {project}/bindings/python/tests"`
- `test-skip = ""` (no skip yet)
- `manylinux-*image = "manylinux2014"`

#### C ABI archive

- `crates/nirs4all-formats-capi/Cargo.toml` gets
  `[build-dependencies] cbindgen = "0.27"`;
- new `crates/nirs4all-formats-capi/build.rs` invokes
  `cbindgen::Builder::new()` to generate
  `crates/nirs4all-formats-capi/include/nirs4all_formats.h`;
- `cbindgen.toml` configures C output with documented prefixes (`n4fmt_`
  symbol prefix, `N4FMT_` macro prefix);
- release job builds `libnirs4all_formats_capi.{a,so,dylib,dll}` per OS and
  uploads `nirs4all-formats-capi-<ver>-<target>.tar.gz` containing the
  static + shared libs, the header, the `LICENSE`, and a `README.md` with
  ABI usage notes.

#### R source release

- `bindings/r/nirs4allformats` source tarball built via `R CMD build`;
- attached as a Release artifact alongside Linux + macOS pre-built
  binaries built via `R CMD INSTALL --build`; no CRAN submission yet
  (deferred per `ROADMAP.md`).

#### Documentation

- new `docs/RELEASE.md` covering the tag → release flow, the dry-run
  procedure on TestPyPI, the rollback story (yanking a wheel), and the
  per-wheel capability matrix.
- `docs/VERSIONING.md` updated to describe how the C ABI version
  (`N4FMT_ABI_VERSION`) bumps independently from the Rust semver.

### Tests vs real data

The release workflow has a `workflow_dispatch` dry-run mode that:

1. Builds wheels on every OS.
2. Installs each wheel into a fresh venv.
3. Runs `python -m pytest bindings/python/tests` against the installed
   wheel — i.e. validates every wheel produces the same records on the
   committed fixtures as the source build.
4. Tar-extracts the C ABI archive and compiles
   `crates/nirs4all-formats-capi/examples/probe_version.c` against it (a tiny
   "open the ABI, print the version" smoke test that we add as part of
   this milestone).

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| `Hdf5File::from_bytes` performance overhead for large fixtures | Benchmark the new path vs the legacy `Hdf5File::open(path)` on `eigenvector_corn.mat` (1.4 MB) and `eigenvector_nir_shootout_2002.mat` (6.6 MB). If the slowdown is significant, keep `read_path` on the disk-backed `Hdf5File::open` and only use `from_bytes` from the resolver path. |
| `opusreader2` GPL-3 isolation | Subprocess only; never imported into the test process; documented in `docs/CONFORMANCE.md`. |
| `cibuildwheel` on macOS arm64 with future C-backed features | Document a capability matrix in `docs/RELEASE.md`; downgrade only on a documented per-OS failure, not preventively. |
| Goldens regenerate cascade in M3 then M2 | Sequence is M3 before M2 precisely to avoid this; goldens are accepted once after M3 and the conformance harness compares against the accepted state. |
| Conformance reader axis orientation drift | Per-format normalization in `tests/conformance/refreaders/*` is the only place that re-orients; tolerances documented per format. |
